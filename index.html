TSimulus

Origins

  • EAM-SDI: research project with Guardis.
  • Specific area of ​​automatic infrastructure management.
  • Collect data of cpus, fans, ...
  • Monitoring, take reactions against events.
  • Problem: lack of real data and in real time.
  • The solution was to simulate a large number of sensors gathering information from a collection and analysis system.

TSimulus

What is TSimulus?

TSimulus (formally, RTS-Gen) is a tool for generating random, yet realistic, time series values.

Time series?
  • An ordered sequence of points in time, each of them is associated with at most one value.
  • Timestamp → value (typically, real).
  • Used in a wide variety of fields.

Random but realistic?

  • Convincing sound.
  • Models, patterns.

TSimulus

How TSimulus works?

In input

A DSL that describes "the look" of the timeseries

Cycles (daily, weekly)

Noise (white, ARMA)

combinations

conditions

Output

triplets (series, timestamp, value)

Load Testing

Kind of Performance Testing which determines a system's performance under real-life load conditions.

This type of testing helps determine how the application behaves when multiple users access it simultaneously or when there are usage spikes.

  • feed the system the largest task it can handle
  • increase the load
  • simulate virtual users
  • expose defects
  • determine the upper limit for all components

Load Testing

Locust

  • Scalable user load testing tool written in Python.
  • Simple-to-use, distributed, user load testing tool and used to help load test web sites or other applications. It can also help you figure out how many concurrent users a system can handle. It can define the behaviour you want for each of your Locust instances. It also gives you the ability to monitor the swarming process from a web UI in real time. Features of Locust are the ability to create your test scenarios using straight Python, to scale a number of users you need to emulate, a nice Web-based UI, is good for testing APIs.

Sources: Locust, Locust Dashboard Image , Locust Charts Image.

Load Testing

Gatling

  • Stress tool built on Scala, Akka and Netty.
  • Allows you to test and measure your application’s end-to-end performance. (Features: It has a simple yet powerful DSL, Scala, Scenario recorder.) Gatling generates web-based visualisations of test results automatically.

Sources: Gatling, Gatling Dashboard Image , Gatling Charts Image.

Load Testing

Tsung

  • Open-source, multi-protocol distributed load testing tool.
  • It can monitor a client’s CPU, memory, and network traffic. (Features: HTTP recorder, HTML reports and graphs, scenarios written in XML)

Sources: Tsung, Tsung Dashboard Image , Tsung Charts Image.

Load Testing

JMeter

  • "most" popular open-source tool in the load testing space.
  • Apache JMeter describes itself as a Java application that was designed specifically for load testing with the ability to measure performance. (performance test a host of technologies such as Java Objects, Web HTTP/HTTPS, SOAP and Rest Services, FTP, Databases with JDBC, IDE you can use for recording, building and debugging your performance tests, Starting with JMeter 3.1 Groovy is the default programming language, Scenarios written in XML…). But after using it in few projects, it appeared that there are some drawbacks: the gui, xml, …

Sources: JMeter, JMeter Dashboard Image.

Taurus

  • Automation-friendly framework for Continuous Testing. It can be seen as an abstraction layer on top of the technologies described here before.
  • The power of Taurus is that it allows you to write your tests in YAML. You can actually describe a full-blown script in about ten lines of text, which gives teams the ability to describe their tests in a YAML or JSON file. (written in YAML, fits performance testing into your CI/CD pipeline, Abstraction layer on top of Locust, Gatling, JMeter, ... and also Selenium, Interesting framework for CI/CD).

Sources: Taurus.

Monitoring

Monitoring is the process of gathering hardware and software measures and ensuring that the service is working as intended.

Monitoring is done by means of checks on average response times, error and request rates, CPU / RAM / ... usage and application availability.

A complete monitoring system will provide a dashboard to check the system health, usage, and send alerts when something is not running properly. It can also be extended to react to changes in the system state (scale the backend when under high load, restart an unresponsive service, ...).

Monitoring the platform consists to check the health of the containers and the health of the orchestrator (Kubernetes). The establishment of mechanisms for monitoring the proper functioning of the solution at several levels of the solution is recommended.

Monitoring

Infrastructure Monitoring

In order to monitor the state of the infrastructure (such as the CPU / memory usage of the servers, the disk space, ...), a tool like Zabbix, Nagios or Icinga is recommended.

Cloud providers have also their own monitoring tools. For instance, AWS provides the Cloudwatch service or Google Cloud provides the Stackdriver service that fulfill this role.

Sources: Nagios, Zabbix, Icinga.

Service Monitoring

Service monitoring via a Service Mesh, a dedicated infrastructure layer that ensures communication between services.

It is implemented as an array of lightweight proxies that are deployed on top of applications.

Service mesh software handles routing, load balancing, provides logging, telemetry, etc.

Istio is a new Microservice service mesh manager for making microservice deployments less complex and eases the strain on development teams. It is a completely open source service mesh that layers transparently onto existing distributed applications. It is also a platform which includes APIs that let it integrate into any logging platform, telemetry or policy system.

Source: Istio.

Monitoring

Service Status

In order to monitor in real time the availability of the Big Data Platform, an external monitoring service must be set up. This service regularly polls the Big Data Platform and notifies (by email, SMS, ...) the system administrators in case of unavailability. A tool like UptimeRobot offers such a service (free basic formula), as well as a status dashboard.

Example of use of the free UptimeRobot service at CETIC.

Source: UptimeRobot.

Monitoring

Management of application logs

In order to easily monitor and consult the application logs (errors, warnings, debug), to audit / trace the actions of the users a posteriori (for example in case of audit), it is recommended to set up a tool for log management to centralize logs of the Big Data Platform components and other system components. The ELK application stack: ElasticSearch, Logstash, Kibana can be integrated with the Big Data Platform. Others tools provided by Cloud providers can also do the same work sush as Cloudwatch from AWS or Stackdriver from Google Cloud.

Elasticsearch is a search engine based on the Lucene library. It provides a distributed, multitenant-capable full-text search engine with an HTTP web interface and schema-free JSON documents. Elasticsearch is developed in Java and is released as open source under the terms of the Apache License.

Logstash is an open source, server-side data processing pipeline that ingests data (e.g. log records) from a multitude of sources simultaneously, transforms it, and then sends it.

Kibana is an open source data visualization plugin for Elasticsearch.
