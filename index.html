#### TSimulus, c’est quoi? TSimulus (formally, RTS-Gen) is a tool for generating random, yet realistic, time series values. Série chronologique? séquence ordonnée de points dans le temps, chacun d'entre eux étant associé à au plus une valeur. timestamp → valeur (typiquement, réel) utilisée dans une grande variété de domaines. Aléatoires, mais réalistes? bruit convaincant modèles, patterns
#### Fonctionnement * TSimulus - An opensource realistic time series generator En entrée: Un DSL pour décrire « l’allure » des séries Cycles (quotidiens, hebdomadaires) Bruit (blanc, ARMA) Combinaisons Conditions Add JSON En sortie: triplets (série, timestamp, valeur) Add Image

## TSAAS There is also an other work that is in progress at CETIC: TSimulus As A Service (TSAAS) where the project aims at building a REST API in front of the TSimulus framework, and a set of configurable websocket routes to consume the TSimulus stream. The backend implements the core logic for the stream handling. It includes - wrapping the TSimulus library for generating streams, - exposing a configuration and control REST API, - exposing websocket routes fed by outputs of the generator - handling the timing of data injection into the websockets. - processing templates for custom rendering of the generated data.
## Load Testing Kind of Performance Testing which determines a system's performance under real-life load conditions. This type of testing helps determine how the application behaves when multiple users access it simultaneously or when there are usage spikes.
#### Locust * https://github.com/locustio/locust (8,889*,last release 0.9.0 September 2018) * Locust is a scalable user load testing tool written in Python. * Locust is a simple-to-use, distributed, user load testing tool and used to help load test web sites or other applications. It can also help you figure out how many concurrent users a system can handle. It can define the behaviour you want for each of your Locust instances. It also gives you the ability to monitor the swarming process from a web UI in real time. Features of Locust are the ability to create your test scenarios using straight Python, to scale a number of users you need to emulate, a nice Web-based UI, is good for testing APIs. image: https://locust.io/ , https://www.saashub.com/locust-alternatives , https://github.com/locustio/locust/issues/688
Gatling: https://github.com/gatling/gatling (3,967* last release 14 Apr 2016). Gatling is a stress tool built on Scala, Akka and Netty. It allows you to test and measure your application’s end-to-end performance. (Features: It has a simple yet powerful DSL, Scala, Scenario recorder.) Gatling generates web-based visualisations of test results automatically. image: https://gatling.io/ , https://gatling.io/docs/2.2/general/reports/ , https://gatling.io/docs/2.3/http/recorder/
Tsung: https://github.com/processone/tsung (1,728* last release 29 Aug 2017). Tsung is an open-source, multi-protocol distributed load testing tool. It can monitor a client’s CPU, memory, and network traffic. (Features: HTTP recorder, HTML reports and graphs, scenarios written in XML) image: http://tsung.erlang-projects.org/2015/07/tsung-1.6.0-released/
JMeter: http://jmeter.apache.org/ seems to be the "most" popular open-source tool in the load testing space. Apache JMeter describes itself as a Java application that was designed specifically for load testing with the ability to measure performance. (performance test a host of technologies such as Java Objects, Web HTTP/HTTPS, SOAP and Rest Services, FTP, Databases with JDBC, IDE you can use for recording, building and debugging your performance tests, Starting with JMeter 3.1 Groovy is the default programming language, Scenarios written in XML…). But after using it in few projects, it appeared that there are some drawbacks: the gui, xml, … image: https://sqa.stackexchange.com/questions/16916/performance-testing-analysing-graph-results-in-jmeter
Taurus https://github.com/Blazemeter/taurus (819*, last release 1.13.2 December 2018) is an Automation-friendly framework for Continuous Testing. It can be seen as an abstraction layer on top of the technologies described here before. The power of Taurus is that it allows you to write your tests in YAML. You can actually describe a full-blown script in about ten lines of text, which gives teams the ability to describe their tests in a YAML or JSON file. (written in YAML, fits performance testing into your CI/CD pipeline, Abstraction layer on top of Locust, Gatling, JMeter, ... and also Selenium, Interesting framework for CI/CD). image: https://github.com/Blazemeter/taurus

Monitoring

Monitoring is the process of gathering hardware and software measures and ensuring that the service is working as intended.

Monitoring is done by means of checks on average response times, error and request rates, CPU / RAM / ... usage and application availability.

A complete monitoring system will provide a dashboard to check the system health, usage, and send alerts when something is not running properly. It can also be extended to react to changes in the system state (scale the backend when under high load, restart an unresponsive service, ...).

Monitoring the platform consists to check the health of the containers and the health of the orchestrator (Kubernetes). The establishment of mechanisms for monitoring the proper functioning of the solution at several levels of the solution is recommended.

Monitoring

Infrastructure Monitoring

In order to monitor the state of the infrastructure (such as the CPU / memory usage of the servers, the disk space, ...), a tool like Zabbix, Nagios or Icinga is recommended.

Cloud providers have also their own monitoring tools. For instance, AWS provides the Cloudwatch service or Google Cloud provides the Stackdriver service that fulfill this role.

Sources: Nagios, Zabbix, Icinga.

Service Monitoring

Service monitoring via a Service Mesh, a dedicated infrastructure layer that ensures communication between services.

It is implemented as an array of lightweight proxies that are deployed on top of applications.

Service mesh software handles routing, load balancing, provides logging, telemetry, etc.

Istio is a new Microservice service mesh manager for making microservice deployments less complex and eases the strain on development teams. It is a completely open source service mesh that layers transparently onto existing distributed applications. It is also a platform which includes APIs that let it integrate into any logging platform, telemetry or policy system.

Source: Istio.

Service Status

In order to monitor in real time the availability of the Big Data Platform, an external monitoring service must be set up. This service regularly polls the Big Data Platform and notifies (by email, SMS, ...) the system administrators in case of unavailability. A tool like UptimeRobot offers such a service (free basic formula), as well as a status dashboard.

Example of use of the free UptimeRobot service at CETIC.

Source: UptimeRobot.

Monitoring

Management of application logs

In order to easily monitor and consult the application logs (errors, warnings, debug), to audit / trace the actions of the users a posteriori (for example in case of audit), it is recommended to set up a tool for log management to centralize logs of the Big Data Platform components and other system components. The ELK application stack: ElasticSearch, Logstash, Kibana can be integrated with the Big Data Platform. Others tools provided by Cloud providers can also do the same work sush as Cloudwatch from AWS or Stackdriver from Google Cloud.

Elasticsearch is a search engine based on the Lucene library. It provides a distributed, multitenant-capable full-text search engine with an HTTP web interface and schema-free JSON documents. Elasticsearch is developed in Java and is released as open source under the terms of the Apache License.

Logstash is an open source, server-side data processing pipeline that ingests data (e.g. log records) from a multitude of sources simultaneously, transforms it, and then sends it.
